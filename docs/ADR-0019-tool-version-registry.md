# ADR-0019: Centralized Tool Version Registry

## Status

Accepted

## Date

2026-03-21

## Context

DevKit manages CI workflow templates for Go, Node.js, Rust, and .NET projects.
These templates reference GitHub Actions (e.g., `actions/checkout@v4`,
`golangci/golangci-lint-action@v7`) and runtime versions (Node 22, pnpm 10,
Go toolchain). Version pins were previously:

- Hardcoded directly in `project-templates/*.yml` with no single source of truth
- Inconsistent across templates (same action pinned at different versions)
- Impossible to update atomically (change in 15+ files for one version bump)
- Un-auditable (no way to check if a project's CI uses a deprecated version)
- Undocumented (no record of why a version was pinned or when it last changed)

When LLMs generate template suggestions, they frequently hallucinate
non-existent versions (e.g., `actions/checkout@v6`, `actions/upload-artifact@v7`)
that pass review but fail at CI runtime. There was no guardrail against this.

## Decision

### 1. Central Registry (`tool-registry.json`)

A single `tool-registry.json` at the repository root is the authoritative
source of truth for all tool version pins. Each entry records:

- `id`: stable key used as the canonical identifier
- `action`: the GitHub Actions `owner/repo` path (null for runtime versions)
- `current`: the version to use in new templates
- `previous`: the last version (enables rollback without external lookup)
- `min_supported`: the minimum acceptable version in any managed project
- `token`: the substitution token used in `src/` templates (`{{TOKEN}}`)
- `changelog_url`: where to check for upstream releases
- `pinned_reason`: non-null if intentionally held below latest
- `notes`: context for unusual decisions

### 2. Tokenized Source Templates (`project-templates/src/`)

All CI templates are maintained as tokenized source files in `project-templates/src/`.
Tokens use `{{UPPER_CASE_TOKEN}}` syntax matching the `token` field in the registry.

```yaml
# src/ci.yml (source — not deployed directly)
- uses: actions/checkout@{{ACTIONS_CHECKOUT_VERSION}}
- uses: golangci/golangci-lint-action@{{GOLANGCI_LINT_ACTION_VERSION}}
```

The rendered outputs in `project-templates/` are generated by `Invoke-VersionUpdate.ps1
-Mode Render` and committed. This means the repo always has ready-to-copy rendered
templates, not token-laden source files that require processing before use.

### 3. Version Update Script (`scripts/Invoke-VersionUpdate.ps1`)

Five operating modes:

| Mode | Purpose |
|------|---------|
| `Status` | Print all tools with current, previous, and min_supported versions |
| `Render` | Apply registry values to all `src/` templates → `project-templates/` |
| `Bump` | Update a tool's version in the registry, then re-render |
| `Rollback` | Swap current ↔ previous for a tool, then re-render |
| `Propagate` | Scan DevSpace projects and open PRs to update outdated action versions |

Bump and Rollback always call Render afterward to keep rendered templates in sync.

### 4. CI Drift Detection (`scripts/check-registry-drift.py`)

A Python script runs in the DevKit CI pipeline (lint.yml) to verify that
`project-templates/*.yml` match what `src/*.yml` would render using current
registry values. Any mismatch is reported as a CI error with GitHub Actions
annotation format.

This prevents:

- Manual edits to rendered templates that bypass the registry
- Rendered templates drifting from their source after a registry update without re-render

### 5. Conformance Check #21 (Tool Version Currency)

The conformance audit checklist includes check #21, which verifies that all
workflow files in a managed project use action versions at or above `min_supported`.
Files below `current` but above `min_supported` generate warnings (soft), not
failures (hard). Floating refs (`@master`, `@latest`) are always a hard fail.

Fix path: `Invoke-VersionUpdate.ps1 -Mode Propagate -Projects <name>` opens
a PR with corrected versions.

## Consequences

### Positive

- Single change (registry + render) updates all templates atomically
- `previous` field enables one-command rollback without git history lookup
- `min_supported` enables automated compliance checking across all projects
- `pinned_reason` creates an audit trail for deliberate holds below latest
- LLM-generated version hallucinations are caught by CI drift detection
- `notes` field captures non-obvious decisions (e.g., "v6 does NOT exist")

### Negative

- Rendered templates are committed twice (source + rendered), adding repo
  size marginally (~5-10 KB for current template count)
- Developers must remember to run `-Mode Render` after manually editing the registry
  (mitigated by CI drift check catching mismatches within one CI cycle)
- `Propagate` mode creates PRs across all projects; noisy if many projects
  are behind (intentional — each PR is an auditable version bump)

### Neutral

- Tokenized source templates are not valid YAML/HCL until rendered; editors
  may flag token syntax. This is expected and acceptable.

## Alternatives Considered

**Dependabot for each project**: Would require Dependabot configuration in
every managed project. Does not solve the template-generation hallucination
problem. Creates per-project noise rather than centralized control.

**Renovate**: Similar to Dependabot; adds per-project config overhead.
Does not integrate with DevKit's template generation workflow.

**Git submodules for templates**: Overcomplicated version management for
CI YAML files. Adds git complexity for marginal benefit.
